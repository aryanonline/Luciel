# Luciel — Drifts (Open and Resolved)

**What this document is:** The single ledger of every gap between the design (`ARCHITECTURE.md` + `CANONICAL_RECAP.md`) and the implementation (this repository, and the running production environment in AWS `ca-central-1`). Each gap is a token with a stable id, a status, and a resolution path.

**What this document is not:** A changelog. A version-history dump. A historical narrative. Closed drifts stay in the doc with a strikethrough so the audit chain of decisions remains walkable, but the document does not accumulate sediment beyond that.

**Maintenance protocol:** Surgical edits only. New drifts are appended with a unique id. Closed drifts get the closing commit/tag noted and the heading wrapped in strikethrough (`~~`). Reopened drifts get a new id; the original stays closed.

---

## Section 1 — Status legend and id scheme

**Status values:**

- `OPEN` — gap exists; no fix in flight
- `IN-PROGRESS` — fix is in flight on a named branch or PR
- `RESOLVED` — fix has landed on `main`; closing commit/tag noted; heading wrapped in `~~strikethrough~~`
- `DEFERRED` — gap acknowledged, deliberate decision to not fix this cycle; rationale recorded inline
- `WONTFIX` — gap re-examined and judged not a drift (design changed, or claim was wrong); rationale recorded inline

**Implementation marker scheme** (used in `ARCHITECTURE.md` and `CANONICAL_RECAP.md` once Phase 2 reconciliation begins applying them):

- ✅ Implemented — repo and prod both match the design
- 🔧 Partial — repo or prod implements some of the design; gaps tracked as drifts here
- 📋 Planned — design committed, no implementation yet; tracked as drift here
- 🔬 Decision-gate — design says "we will choose later"; not a drift, an open product decision

**Drift id scheme:** `D-<short-slug>-YYYY-MM-DD`. The date is when the drift was first opened, not when it was discovered later. Slug is short, hyphenated, and stable — never rename a slug after creation.

**Source of truth:** When this document, the architecture document, the canonical recap, the repository, or production disagree, the resolution is recorded here as a drift. The four artifacts converge by closing drifts, not by editing one and hoping the others catch up.

### D-celery-beat-single-replica-coupling-2026-05-14

**Status:** OPEN — future-debt. Stays open beyond Step 30a.2. Acceptable at current scale (one paying customer, one worker replica); revisit when worker horizontal scaling becomes load-bearing.
**Raised by:** Computer (advisor) on 2026-05-14 during the Step 30a.2 retention-worker deployment review. The Step 30a.2 design ships the Celery beat scheduler embedded in the same container as the worker via the `--beat` flag (Dockerfile.worker injects `celery -A app.worker.celery_app worker --beat --loglevel=info --concurrency=2`). Beat is what enqueues the 08:00 UTC nightly `run_retention_purge` task; the worker pool is what consumes it. Co-locating beat with the worker means running >1 worker replica would duplicate every scheduled tick — two replicas would enqueue two retention purges at 08:00 UTC every night, three replicas would enqueue three, and Celery's lock semantics on the retention task would absorb the duplication safely (idempotency cursor on `tenants.deactivated_at` plus the leaf-first delete order) but the duplicate work would still hit the database. Distinct from `D-celery-worker-runs-as-root-2026-05-11` (which is the root-user gap, not the beat-coupling gap).
**Design claim:** ARCHITECTURE §3.2.4 (Background worker tier) commits to "a separate pool of worker processes" with "worker autoscaling on queue depth and CPU" — autoscaling implies >1 replica is a valid operational state. The Step 30a.2 single-replica coupling is therefore a temporary constraint, not a design decision: the design says workers scale horizontally, the current implementation says scheduled tasks fire once *per replica* rather than once *per cluster*. Closing the gap requires either (a) splitting beat into its own single-replica ECS service with the worker pool free to scale, or (b) adding a Celery beat lock (e.g. `celery-redbeat` against the existing SSM `REDIS_URL` for dev or against the prod SQS broker for live) that coordinates ticks across multiple beat-embedded workers. Option (a) is the cleaner architectural shape; option (b) preserves the single-container deploy and is reversible.
**Repo state at the time of raise:** `app/worker/celery_app.py` carries the Step 30a.2 beat schedule (`beat_schedule = {'retention-purge-nightly': {'task': 'app.worker.tasks.retention.run_retention_purge', 'schedule': crontab(hour=8, minute=0)}}`). `Dockerfile.worker` injects `--beat`. `cfn/luciel-prod-worker-autoscaling.yaml` exists (and is presumed deployed per `D-prod-worker-autoscaling-unverified-2026-05-09`) — the autoscaling stack's min-capacity is 1 and max-capacity is the GTM-burst ceiling; setting min-capacity > 1 today would activate the beat-duplication failure mode. No celery-redbeat dependency; no separate beat service in CFN.
**Operational impact today:** Zero. Worker desired-count is 1 today and stays at 1 through any plausible single-paying-customer load. The gap activates only when (a) worker queue depth or CPU triggers the autoscaling policy to scale out, OR (b) an operator manually raises the worker service's desired-count above 1 for any reason (e.g., during a deploy when the new task definition is being rolled out and old + new revisions briefly co-exist). The autoscaling policy in `cfn/luciel-prod-worker-autoscaling.yaml` keeps min-capacity at 1, so the deploy-window co-existence is the more likely activation path. Even when activated, the retention task is idempotent (the cursor + leaf-first order absorb duplicate ticks) — the impact is duplicate database load at 08:00 UTC nightly, not a correctness failure.
**Owning roadmap step:** None today. The closure is a Pattern E hardening commit that lands when worker horizontal scaling becomes load-bearing — either when the GTM plan's burst projection exceeds single-replica capacity, or when a second scheduled task lands whose duplication semantics are not idempotent like retention is. Likely natural home: alongside Step 33b (Dedicated infrastructure tier) or as a side-observation commit during Step 34 (Workflow actions, which adds the second wave of scheduled background tasks). Not a roadmap row of its own — the design is correct, the implementation is a known-tactical compromise.
**Resolution path:** Two viable options when this closes. (a) **Separate beat service.** Add `cfn/luciel-prod-beat.yaml` provisioning a single-replica ECS service running `celery -A app.worker.celery_app beat --loglevel=info` (no `--beat` flag on the worker service; the worker pool runs `celery worker` only). Rolls forward via Pattern E: the new beat service comes up alongside the existing `--beat`-embedded workers, then the worker task-definition revision drops `--beat`, then the embedded beat stops firing as old tasks drain. Operational acceptance: worker service min-capacity raises from 1 to 2 (or whatever the autoscaling floor needs); the 08:00 UTC retention tick fires exactly once per night regardless of worker count. (b) **celery-redbeat lock.** Add `celery-redbeat` to `requirements.txt`; configure the beat backend in `app/worker/celery_app.py` to use Redis (or the existing prod SQS broker as a leader-election surface); workers keep the `--beat` flag, but only the lock-winning worker fires ticks. Lighter operational footprint, more coupling complexity. The choice between (a) and (b) is the closure-time decision; this drift records both so the next operator does not re-discover the option space.
**Pattern E:** No row mutation. The closing commit will be additive (new service or new dependency) and the deactivation of the embedded `--beat` will land as a task-definition revision shift, not as a destructive container-config edit. No customer-visible state changes.

## Section 6 — How drifts are added, closed, or reopened

**Adding a drift.** Append to Section 3 (open) or Section 4 (production-pending), with a fresh `D-<slug>-YYYY-MM-DD` id, a status, the section that raised it, the design claim, the repo or prod state, and a resolution path. Do not add to closed-drift section while the drift is still open.

**Closing a drift.** Move the entry verbatim to Section 5, wrap the heading in `~~strikethrough~~`, fill in the closing commit (and tag if relevant) and a one-line resolution. Do not delete the entry from Section 3 — move it. The audit chain depends on the body of each drift remaining readable after closure.

**Reopening a drift.** Open a new drift with a fresh id and a date suffix matching the day of reopening. Reference the closed predecessor in the body. The closed predecessor stays closed; reopening is a new story.

**Marker propagation.** As drifts close, the corresponding section in `ARCHITECTURE.md` or `CANONICAL_RECAP.md` gets the appropriate ✅ / 🔧 / 📋 / 🔬 marker added in surgical edits. The marker is the connective tissue between the design docs and this register; the doc edit and the drift closure are part of the same commit.

**What does NOT belong in the design docs.** The three canonical docs (`CANONICAL_RECAP.md`, `ARCHITECTURE.md`, `DRIFTS.md`) are stable references, not running notebooks. Build-time detail — framework choice, library selection, schema column lists, transport-protocol rationale, branding-knob inventories, sandbox demo plans — belongs in **commit messages** and **code-level docstrings**, not in the canonical docs. The canonical docs grow only when the architectural shape changes: a new cross-cutting property, a new component, a deletion of something that is no longer true, or a roadmap step landing such that its success criterion is provably met. A new roadmap step, on its own, is not a reason to grow the docs — the recap row already exists, and implementation detail is recoverable from `git log` and the code itself. If a build genuinely needs a multi-day shared scratchpad, it lives in `docs/in-flight/<slug>.md` during the build and is **deleted** (not archived) on merge. This principle exists because docs that grow with every roadmap step become unparsable; the discipline of keeping them stable is what makes them useful as references.
