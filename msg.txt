SCOPE NOTE

This commit ships the *infrastructure* for prod migrations (working
luciel-migrate:11 task-def, Pattern N codified, runbook updated). The
actual application of pending migrations (Commit 7's f392a842f885 and
others between prod head and Commit 7) is deferred to Commit
8b-prereq-data, because the first invocation of luciel-migrate:11 today
surfaced prod-state drift requiring a backfill decision (see RECON below).

WHY (Phase 1 D-luciel-migrate-task-def-hollow-since-rev1,
     D-undocumented-laptop-migrate-path)

Recon during Commit 8b execution discovered that luciel-migrate task-def
had command: null since revision 1. All prior production migrations were
therefore applied via path (i): operator-laptop direct connection to
prod RDS using `alembic upgrade head` with DATABASE_URL pointing at prod,
sourced from SSM at runtime. This path was undocumented in the repo and
contradicted Pattern E's discipline (raw secrets only ever in SSM
SecureString, never in stdout/CloudWatch/git).

Three structural drift items resolve in this commit:

- D-luciel-migrate-task-def-hollow-since-rev1: register :11 with command
  override, with image rebuilt from current branch state.
- D-undocumented-laptop-migrate-path: codify Pattern N as the canonical
  prod-migration path. Operator-to-prod-RDS connections are
  architecturally bounded to ECS one-shot tasks running in the VPC.
- D-runbook-c3-circular-dependency: Phase C.3 referenced a laptop-IP
  allowlist that doesn't exist; replaced with Option alpha verification
  chain.

Plus three runbook authoring-quality fixes batched in:

- D-runbook-jmespath-typo: 5+ instances of `Foo` to `Foo[0]` swept across
  Phase A, Phase B, Phase C verify queries.
- D-runbook-em-dash-in-aws-descriptions: A.4 rule 2 description fixed
  (em-dash to ASCII hyphen). Generalized to D-runbook-ascii-only-for-
  tooling: all runbook content destined for AWS descriptions, JMESPath,
  JSON files, or any tooling pipeline must be authored in ASCII.
- D-pattern-n-prod-migrations: new pattern codified.

Brokerage tech-due-diligence posture: "How do you apply database
migrations to production?" now has a single, defensible answer documented
in source control at docs/runbooks/operator-patterns.md.

FIRST-RUN FINDING (motivates Commit 8b-prereq-data)

First invocation of luciel-migrate:11 against prod RDS exited with code 1
on migration 4e4c2c0fb572 (D11 memory_items.actor_user_id NOT NULL flip),
with psycopg.errors.NotNullViolation: column "actor_user_id" of relation
"memory_items" contains null values. Surfaced two facts:

- Prod RDS alembic_version is at 4e989b9392c0 (pre-D11). Local repo head
  is at f392a842f885 (Commit 7). Prod is behind local main by 3 migrations.
- Prior 10 production migrations applied via the operator-laptop path
  stopped before D11. The recap section 5.2 line listing D11 (baf06f7),
  D5 (67c65a5), and D-worker-role (40d9fb8) as "RESOLVED" referred to
  *local verification suite passes*, not prod application.

Alembic's transactional DDL discipline rolled the failed transaction back
cleanly. Prod RDS state is unchanged from before this commit (still at
4e989b9392c0). No data damage, no partial role creation, no manual cleanup.

Resolution path: Commit 8b-prereq-data will codify Pattern O (read-only
prod queries via ECS one-shot task), use it to count and sample the
NULL-actor_user_id rows in prod, and ship a backfill migration to bring
prod to current head. Phase 1 grows by one more commit; close shifts to
late today / tomorrow.
