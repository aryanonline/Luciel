WHY

The original Commit 8b-prereq-data scope was "fix the migrate path
followed up immediately by applying pending migrations to prod RDS."
Commit 8b-prereq (this morning) handled the migrate-path fix and
codified Pattern N. The data-apply portion was the planned content
of this commit.

Pre-mutation recon via the newly-codified Pattern O surfaced a
structural drift item that fundamentally reframed the work. Specifically:

- 7 recon queries against prod RDS via `luciel-recon:1`, totaling
  ~$0.005 USD compute and ~7 minutes wall clock.
- Discovered prod RDS contains 29 tenants in `tenant_configs`, of
  which 100% match verification-tenant naming patterns
  (`step24-5b-*`, `step26-verify-*`, `step27-prodgate-*`,
  `step27-syncverify-*`).
- Discovered the single NULL `actor_user_id` row in `memory_items`
  belongs to `step27-syncverify-7064`, with all four parent-chain
  rows (tenant_configs, luciel_instances, sessions, messages) still
  present in prod.
- Discovered prod `alembic_version` head is `4e989b9392c0` (matches
  recap claim).

The first finding is the structurally significant one. The verification
suite's Pillar 10 ("teardown integrity, zero residue") is suite-internal
and runs against dev. Prior verification flows that ran during prod
deploys (Step 24.5b prod gate, Step 26b/c verifies, Step 27c-final
sync verify) had no equivalent teardown contract for prod, and their
synthetic tenant residue has accumulated since 2026-04-22.

Applying the NOT NULL flip on top of 29 verification residue tenants
would technically work but produce a brokerage-DD-indefensible audit
state ("you applied a constraint to data you knew was test residue").
The right sequencing is:

  1. Codify Pattern O. (this commit)
  2. Clean up the 29 verification residue tenants via the existing
     admin DELETE /tenants/{id} code path (next commit).
  3. Apply pending migrations against the now-clean prod RDS.
  4. Resume Commit 8b's worker role swap.

Today's work scopes to (1) plus the discovery itself. Steps 2-4 are
deliberate next-session work, not autopilot continuation today.

The recon work itself worked exactly as Pattern O is designed to:
zero prod state changes, full three-channel audit, all queries
captured in CloudWatch and CloudTrail with deterministic SHA256
correlation. Pattern O earned its codification by catching the
issue at the cheapest possible gate.
